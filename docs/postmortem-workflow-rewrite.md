# 自研工作流重写失败复盘

日期：2026-06-10  
项目：模镜 ModelMirror  
范围：工作流引擎与 RAG 资料库从 Dify 集成模式切换到纯自研模式的失败尝试

## 1. 失败症状

- 工作流和资料库页面从稳定的 Dify iframe 集成切换为本地自研页面后，功能边界变得模糊，主入口无法保证稳定可用。
- 多个前端文件出现乱码文案和不完整字符串，影响页面渲染、类型检查和用户可读性。
- 自研资料库只实现了浏览器本地分段和简单关键词检索，远未覆盖 Dify 的文档解析、向量索引、检索测试、应用绑定和运维能力。
- 自研工作台把应用、工作流、发布、日志、设置等多个复杂域一次性堆入同一轮迭代，导致实现深度不足。
- 回退前没有建立明确的 feature flag、并行路由和自动化测试，主路径被实验代码污染。

## 2. 根因分析

### 2.1 需求膨胀

一次性尝试复刻 Dify 的完整能力，包括工作流画布、节点系统、执行引擎、知识库管理、RAG 管道、发布和观测。这些能力本身就是一个成熟平台的核心产品线，远超单次迭代能够安全交付的范围。

### 2.2 缺乏架构设计

在实现前没有先定义稳定的数据模型、节点能力矩阵、前后端 API 契约、执行状态机和错误边界。结果是页面、数据和执行逻辑互相耦合，难以局部替换，也难以安全回滚。

### 2.3 忽视测试

核心执行逻辑没有单元测试，前端页面没有基本渲染测试，资料库检索没有可重复的样例校验。问题只能在手动运行时暴露，且一旦出现乱码、接口不匹配或状态异常，很难快速定位。

### 2.4 一步到位思维

实验代码直接替换了稳定入口，而不是放在 `/workflow-native` 之类的独立路由中与 Dify 并行运行。没有保留稳定 fallback，导致用户体验被未成熟实现影响。

### 2.5 文案与编码治理不足

多处中文文案通过不安全的批量替换或非 UTF-8 控制台路径写入，造成源码层面的 mojibake。说明当前工程缺少编码检查和构建前的文本质量扫描。

## 3. 可执行教训

- 大型功能重构必须拆成多个可独立验收的里程碑，不允许一次性替换稳定主路径。
- 每个里程碑必须有明确的验收标准，包括构建、后端语法检查、接口样例、失败场景和 UI 可用性。
- 工作流执行器、变量解析、拓扑排序、条件分支、RAG 检索等核心逻辑必须先在后端通过单元测试验证。
- 前端自研编辑器必须放在独立路由中，与 Dify iframe 稳定版本并行运行。
- 所有自研数据结构应尽量保持与 Dify 能力概念兼容，便于后续迁移、对照和回退。
- 中文源码写入必须使用 UTF-8，并在提交前运行乱码扫描和构建验证。

## 4. 后续红线

- 禁止在没有设计文档和接口契约的情况下进行大规模工作流重构。
- 禁止实验功能直接替换 `/workflow` 和 `/rag` 稳定入口。
- 禁止在测试套件、前端构建、后端语法检查未通过时交付。
- 禁止把“平台级能力复刻”压缩为单次交付任务。
- 禁止通过不安全批量替换处理中文源码和用户可见文案。

## 5. 当前救援结论

本次救援已将工作流和资料库主路径恢复到 Dify 稳定集成模式。自研工作流保留为备用实验入口，后续必须按独立路线、独立测试、独立验收推进。
